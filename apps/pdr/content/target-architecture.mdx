## 3.2 Why Lightdash

We evaluated the major open-source BI platforms against our core requirements: **code-first dashboards**, **semantic layer**, **Git-native workflows**, **AI-readiness**, **self-hostable**, and **BigQuery compatibility**.

| Requirement                          | [Lightdash](https://www.lightdash.com/)                                      | [Metabase](https://www.metabase.com/) | [Apache Superset](https://superset.apache.org/) | [Evidence](https://evidence.dev/) |
| ------------------------------------ | ---------------------------------------------------------------------------- | ------------------------------------- | ----------------------------------------------- | --------------------------------- |
| **Dashboards as code (YAML in Git)** | ✅ Native                                                                    | ❌ GUI-only                           | ❌ GUI-only                                     | ✅ Markdown-based                 |
| **Semantic layer**                   | ✅ Built-in                                                                  | ❌ None                               | ❌ None                                         | ❌ None                           |
| **Git-native CI/CD**                 | ✅ CLI + GitHub Actions                                                      | ❌ No CLI                             | ❌ No CLI                                       | ✅ Git-native                     |
| **dbt integration**                  | ✅ Designed for dbt                                                          | ❌ Limited                            | ❌ Limited                                      | ⚠️ Partial                        |
| **AI-ready (LLM skills)**            | ✅ [Official skills](https://docs.lightdash.com/guides/ai-overview)          | ❌ None                               | ❌ None                                         | ❌ None                           |
| **Self-serve exploration**           | ✅ [Explore view](https://docs.lightdash.com/guides/exploring-your-content/) | ✅ Question builder                   | ✅ SQL Lab                                      | ❌ Read-only reports              |
| **Self-hostable**                    | ✅ Docker / Render                                                           | ✅ Docker                             | ✅ Docker                                       | ✅ Vercel / Node                  |
| **BigQuery support**                 | ✅                                                                           | ✅                                    | ✅                                              | ✅                                |

**Why Lightdash wins for us:**

- **Only platform that combines dashboards-as-code with a semantic layer.** Metabase and Superset are powerful GUI tools, but dashboards live inside the application — not in Git. You can't version control them, review them in PRs, or generate them with AI.
- **Purpose-built for dbt.** While other tools can connect to a warehouse, Lightdash reads dbt model metadata natively. When we add dbt in Phase 2, the integration is seamless — no adapter layer, no workarounds.
- **Official AI/LLM support.** Lightdash provides agent skills that teach AI tools like Claude Code how to build correct charts and metrics. No other open-source BI tool offers this today.
- **Evidence is close but different.** Evidence produces beautiful static reports from Markdown + SQL, but it lacks a semantic layer, live exploration, and interactive dashboards. It's a reporting tool, not a BI platform.
- **Metabase and Superset are great tools for different needs.** Metabase excels at self-serve for non-technical users. Superset excels at scale and advanced visualizations. Neither supports the code-first, AI-ready workflow we need.

## 3.3 Design Principles

- **Open-source first** — Lightdash, dbt, GitHub Actions. No proprietary platforms, no license fees for core tooling.
- **Modular** — Each component does one job. Any piece can be swapped without rebuilding the rest.
- **Code-first** — Dashboards are YAML files, metrics are SQL definitions, everything lives in Git. Because it's code, it's portable, diffable, reviewable, and reproducible.
- **AI-ready** — Because everything is code, AI tools can read, write, and reason about our analytics stack. This isn't a future aspiration — it's a property of the architecture from day one.
- **Budget-conscious** — Built on open-source tools with low hosting costs. Infrastructure can be scaled up as usage grows, increasing both capacity and capability incrementally.

## 3.4 CI/CD

Every change to a dashboard, chart, or metric definition goes through an automated pipeline:

```text
Author       Push to      CI runs       Preview      Team        Merge to
edits   ---> branch  ---> validation -> env     ---> reviews --> main
YAML         on GitHub    checks        spins up     the PR      |
                                                                 v
                                                            Production
                                                            updates
```

**What CI checks on every change:**

- ✅ All chart field references point to real model definitions
- ✅ YAML parses correctly
- ✅ Dashboard renders without errors in a [preview project](https://docs.lightdash.com/guides/cli/how-to-use-lightdash-preview/)
- ✅ Formatting and linting rules pass

![CI/CD pipeline checks](/poc_ci.png)

> _Today: every change goes directly to production with no validation. With this architecture: every change is validated automatically before anyone sees it._

## 3.5 Dev Environment vs. Prod Environment

|                 | Dev (Preview)                          | Production                 |
| --------------- | -------------------------------------- | -------------------------- |
| **When**        | Created automatically per pull request | Updated on merge to `main` |
| **Who sees it** | The author + reviewers                 | Leadership + all consumers |
| **Purpose**     | Test and review changes safely         | Trusted, stable dashboards |
| **Lifecycle**   | Destroyed when PR is closed            | Always live                |

Today, every change goes directly to production. With this architecture, developers iterate freely in [preview environments](https://docs.lightdash.com/guides/cli/how-to-use-lightdash-preview/) while production remains untouched until changes are explicitly approved.

## 3.6 Dashboards-as-Code

Instead of building dashboards through a point-and-click GUI, every dashboard is defined in a **code-first** YAML file:

```yaml
name: Sales Performance
spaceSlug: revenue/dashboards
tiles:
  - type: saved_chart
    properties:
      chartSlug: bookings-acv
  - type: saved_chart
    properties:
      chartSlug: pipeline-created
```

| Capability              | Web-only GUI (Looker Studio)    | Code-first (Lightdash + Git)                                                                                       |
| ----------------------- | ------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| See what changed        | ❌ No history                   | ✅ Line-by-line diff in every PR                                                                                   |
| Test before deploy      | ❌ Changes are live immediately | ✅ Preview environment + CI validation                                                                             |
| Rebuild from scratch    | ❌ Manual recreation            | ✅ `git clone` and deploy                                                                                          |
| AI-assisted development | ❌ Impossible — no code to read | ✅ Claude Code reads, writes, and modifies YAML                                                                    |
| Reuse across dashboards | ❌ Copy-paste                   | ✅ Shared [chart slugs](https://docs.lightdash.com/guides/cli/how-to-use-lightdash-upload/) and metric definitions |

> _Today: modifying a chart means clicking through a GUI, testing manually, hoping nothing broke. With dashboards-as-code: every change is a reviewable pull request with automated validation._

## 3.8 The Semantic Layer

**The problem: same metric, different numbers.**

When metric definitions live inside individual dashboards, inconsistencies are inevitable. "Revenue" in one dashboard might include renewals. In another, it might not. Both show "Revenue" — but the numbers don't match. This isn't a data quality problem — it's a definition problem.

**The solution: one definition, used everywhere.**

A [semantic layer](https://docs.lightdash.com/guides/how-to-create-metrics/) defines every metric once, in code:

```yaml
metrics:
  bookings_acv:
    type: sum
    sql: ${TABLE}.acv
    label: 'Bookings (ACV)'
    description: 'Total Annual Contract Value of closed-won opportunities'
```

Every dashboard, every ad-hoc query, and every AI agent reads from this single definition. Change it once, it updates everywhere. Review the change in a pull request. See the full history of how a metric evolved over time.

Because metrics are **code-first**, they are automatically **AI-ready** — an AI agent querying your data reads the same definitions your dashboards use.

![Semantic layer in Lightdash Explore view](/poc_semantic_layer.png)

| Capability                                                                              | Without semantic layer                | With semantic layer                           |
| --------------------------------------------------------------------------------------- | ------------------------------------- | --------------------------------------------- |
| **Agentic BI**                                                                          | AI guesses at SQL, unreliable         | AI queries defined metrics, trustworthy       |
| **Self-serve [exploration](https://docs.lightdash.com/guides/exploring-your-content/)** | Users need raw table/column names     | Users browse labeled, documented metrics      |
| **Automated QA**                                                                        | No way to validate metric correctness | CI checks every chart against defined metrics |
| **Cross-dashboard consistency**                                                         | Each dashboard defines its own logic  | All dashboards read from one source           |
| **Onboarding**                                                                          | New analysts reverse-engineer SQL     | New analysts browse a metric catalog          |

## 3.9 The dbt Opportunity

**What dbt is, in business terms:**

[dbt (data build tool)](https://docs.getdbt.com/docs/introduction) is an open-source framework that lets you write SQL transformations as version-controlled, tested, documented code — instead of scheduled queries managed through a web console. Think of it as applying the same **code-first** practices we're proposing for dashboards to the data transformation layer underneath.

**What it unlocks:**

- **Version-controlled transformations** — every SQL change is a pull request, reviewable and reversible
- **Built-in [testing](https://docs.getdbt.com/docs/build/data-tests)** — define expectations ("revenue should never be negative") and catch violations automatically
- **Automatic [documentation](https://docs.getdbt.com/docs/collaborate/documentation)** — dbt generates a browsable catalog of every table, column, and transformation
- **[Lineage tracking](https://docs.getdbt.com/terms/data-lineage)** — see at a glance which dashboards are affected when an upstream table changes
