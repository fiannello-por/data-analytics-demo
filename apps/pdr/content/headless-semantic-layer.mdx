## Introduction

### How We Got Here

The PDR established that we need to modernize off Looker Studio. Lightdash was proposed as the full-stack replacement: semantic layer, dashboards-as-code, CI/CD, AI-readiness, all open source and budget-friendly.

The feedback was that Lightdash&apos;s visualization layer, tile-based and grid-constrained, does not meet the bar for what leadership wants to present to stakeholders. The dashboards need to feel like a product, not a BI tool.

That creates the real tension:

No single tool checks all five boxes. So the question becomes: can we split the stack? Keep Lightdash for what it is great at, semantic layer, governance, self-service exploration, and CI/CD, and use Next.js for what it is great at, pixel-perfect, fully custom visualization.

That is the hypothesis this POC tests.

### Scope

This POC answers one question with one dashboard.

### In Scope

- Lightdash works as a headless semantic layer: its API compiles semantic intent, model + measures + dimensions + filters, into SQL without ever rendering a chart
- A Next.js app can consume that API to render a fully custom dashboard with complete UI freedom
- The governance model survives: metrics are still defined once in YAML, version-controlled, CI-validated, and shared between Lightdash&apos;s own UI and the custom webapp
- Performance is viable: compile + execute latency is acceptable for an interactive dashboard
- One working example: the Sales Performance dashboard, rebuilt in Next.js and powered by Lightdash&apos;s semantic layer

### Out of Scope

- This is not a production architecture recommendation; it is a feasibility test
- This is not a design review of the dashboard UI; the point is to prove that the same data can be reproduced outside Lightdash and that adding Lightdash to the stack does not introduce unacceptable query latency

## How It Works

<ArchitectureOverviewDiagram />

1. **The UI asks the Semantic Layer:** "I need the close rate metric, filtered by expansion customers, grouped by week." This is a semantic intent. It describes what, not how.

2. **The Semantic Layer responds with compiled SQL:** A fully formed query ready to execute, built from version-controlled metric definitions. The UI never writes SQL.

3. **The UI sends the SQL to the Warehouse:** The compiled query goes directly from the UI to BigQuery. The Semantic Layer is not involved in execution.

4. **The Warehouse returns rows:** Raw result data flows back to the UI, which renders it as charts, tables, or any other visualization.

See the live example: [Sales Performance Dashboard](https://analytics-suite.vercel.app/dashboards/sales-performance?category=Expansion&startDate=2026-01-01&endDate=2026-03-27&tileId=expansion_annual_pacing_ytd).
